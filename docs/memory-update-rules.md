# Правила обновления файлов памяти

> Когда поступает новая информация — обновлять файлы сразу, не ждать «запомни это».

## Где что хранится

| Тип информации | Файл | Действие |
|---|---|---|
| Новый контакт (имя, TG id, бот) | `CONTACTS-private.md` | Добавить карточку |
| Новое о существующем контакте | `CONTACTS-private.md` | Дописать в карточку |
| Технический факт о проекте/инстансе | `MEMORY.md` | Добавить блок с датой |
| Событие дня (что обсудили, что сделали) | `memory/YYYY-MM-DD.md` | Дописать в daily |
| Долгосрочное правило/решение | `MEMORY.md` + стилевой файл если про тон | Добавить раздел |
| Сообщение в Telegram | `log-YYYY-MM-DD.jsonl` | Автоматически через прокси |

## Когда обновлять

✅ **Сразу после факта** — не откладывать «на потом», контекст теряется

✅ **Не спрашивать разрешения** на мелкие правки (даты, статусы, новые ID)

✅ **Крупные правки** (новая категория, удаление, переименование) — упомянуть владельцу в ответе

## Что НЕ записывать

❌ Одноразовые вопросы и общие знания
❌ Временные запросы («как сейчас погода», «который час»)
❌ Слишком сырое / не подтверждённое
❌ Дублирование уже записанного

## После крупных обновлений

```bash
# Реиндексация в RAG для семантического поиска
bash scripts/remember.sh "<краткая выжимка факта>"
```

## Пример: добавился новый клиент

1. **Сразу** — карточка в `CONTACTS-private.md` (даже минимальная: имя + TG id)
2. **После звонка/обсуждения** — расширить карточку (продукт, инстанс, проблемы)
3. **При первом инциденте** — добавить ⚠️ с описанием в его карточку
4. **Когда проблема решена** — убрать ⚠️, дописать что сделано
5. **Долгосрочные факты** (типа «использует deepseek через OpenRouter, не напрямую») → в `MEMORY.md`

## Пример: владелец передал новое правило

1. Понять что это: разовая задача или долгосрочное правило?
2. Если долгосрочное → раздел в `MEMORY.md` с датой
3. Если про стиль общения → также добавить в `vika-style.md`
4. Подтвердить владельцу: «Записала в MEMORY.md + vika-style.md»
