# Архитектура GH-Secretary

Двух-ботовая система, выступающая «секретарём-прокси» между владельцем (Роман) и его контактами в Telegram.

## Зачем

Когда владелец занят / спит / в отпуске — секретарь:
- Принимает входящие сообщения от любых контактов через Telegram Business
- Отвечает в стиле и от лица владельца (Вика)
- Эскалирует важное (семья, деньги, юридическое, манипуляции) — сразу в личку владельца
- Если владелец сам ответил в чат — pending-ответ автоматически отменяется

### Зачем два бота

| Бот | Роль | Режим |
|---|---|---|
| `@VikaSecretary_bot` | Принимает входящие в Business-чатах, отвечает «от Романа» | Business Mode (webhook) |
| `@OneInt_bot` | Личка Романа с самой Викой (control plane, отчёты, эскалации) | Polling в OCPlatform |

Telegram Business позволяет боту читать/писать в чатах владельца от его имени. Бот видит ID соединения (`business_connection_id`), но не видит токен пользователя — отправка идёт через специальные методы API.

## Поток сообщения

1. Контакт пишет в чат с Романом → Telegram доставляет событие на webhook
2. `secretary-proxy` сохраняет сообщение, определяет автора (по `from.id`)
3. Если автор — Роман сам → пропускаем (это владелец отвечает себе или контакту)
4. Если автор — контакт:
   - Проверяем `contacts.json` — знакомый или нет?
   - Если **критическое** (триггеры: семья, деньги, встречи, юридическое) → forward в `@OneInt_bot` + НЕ отвечаем сами
   - Иначе → ставим в `pending.json` с таймером **2 минуты днём (08–18 МСК) / 3 минуты ночью (18–08 МСК)**
5. Через таймер:
   - Проверяем не ответил ли Роман сам → если да, отменяем
   - Иначе генерируем ответ через `vika.js` (LLM с системным промптом стиля Вики)
   - Отправляем через Business API от имени Романа
6. Лог события → `log-YYYY-MM-DD.jsonl`

## Стейт (persistence)

`STATE_DIR=/root/.ocplatform/workspace/memory/secretary/`

| Файл | Содержание |
|---|---|
| `connections.json` | Активные business_connection_id |
| `contacts.json` | Известные контакты (имя, теги, контекст) |
| `conversations.json` | История по chat_id (последние N сообщений) |
| `conversations/<chat_id>.jsonl` | Полная история переписки |
| `pending.json` | Очередь отложенных ответов |
| `log-YYYY-MM-DD.jsonl` | Все события за день |
| `tokens.env` | Токены ботов (chmod 600, в `.gitignore`) |

При рестарте все pending восстанавливаются из `pending.json` — отложенные ответы не теряются.

## Эндпоинты

| Метод | Путь | Описание |
|---|---|---|
| `POST` | `/tg/business-webhook` | Telegram Business webhook (снаружи через Nginx: `/secretary/tg/business-webhook`) |
| `POST` | `/api/reply` | Ручной ответ клиенту по mapping_id |
| `GET` | `/api/contacts` | Известные контакты |
| `GET` | `/api/conversations` | Карта разговоров |
| `GET` | `/api/pending` | Текущая очередь отложенных |
| `DELETE` | `/api/pending/:chatId` | Отменить отложенный ответ |
| `GET` | `/health` | Health check |

Все пути `/api/*` требуют заголовок `X-Api-Key` (env `API_KEY`).
