# Отчёт о готовности к продакшну

> Дата: 2026-06-23. Версия: ветка `claude/stoic-gauss-txav4q` (= `main`):
> этапы 0–5 + надёжность + SaaS S1–S5 + шифрование/ротация секретов + BYO-LLM +
> приём оплаты (Robokassa) + личный кабинет.
> Метод: аудит кода + 166 автотестов + живые проверки.

## Два вердикта

**1. Одиночный режим (один владелец): ✅ Production-ready.**
Зрелый, обкатанный путь — все функции работают, надёжность доведена. Это целевой
сценарий шаблона: один человек, один экземпляр, один workspace памяти (в т.ч. OpenClaw).

**2. SaaS для нескольких платящих клиентов: 🟡 Beta — «слой данных» готов, «операционный слой» нет.**
Изоляция, тарифы, биллинг, онбординг, кабинет — построены и протестированы. Но
повседневное управление каждого клиента из Telegram (команды/кнопки, автопостинг,
дайджест) пока работает **глобально на одного владельца** (`default`), а не на каждого
арендатора. Это главный блокер перепродажи — см. «Ограничения» и эпики ниже.

❌ **Не готов к** высокой доступности (несколько реплик одного бота конфликтуют по
`getUpdates`) — масштабирование вертикальное.

## Что готово (по слоям)

### Надёжность / эксплуатация
- Graceful shutdown (SIGTERM/SIGINT: HTTP, поллинг, БД; hard-exit 10с), обработка
  `uncaughtException`/`unhandledRejection`.
- SQLite WAL + `busy_timeout=5с` + `synchronous=NORMAL`; дедупликация вебхуков в БД
  (переживает рестарт).
- Таймауты LLM 45с / Telegram 15с; health со `SELECT 1` к БД (503 при сбое); лимит
  тела; глобальный error-handler. Docker non-root + healthcheck + volume; ротация логов.

### Функции секретаря (личка)
- Отложенный автоответ (2 мин днём / 3 ночью), отмена если владелец ответил сам.
- Управление: `/on /off /vacation /draft /status` + кнопки. Draft-режим.
- Политики `auto/escalate/ignore`. Голос/фото — эскалация/Whisper. Контекст 25 сообщений.
  Петля качества (правки → few-shot, 👍/👎). База фактов. Дайджест владельцу.

### Публичные поверхности и мультиплатформа
- Комментарии канала, Q&A в группах (упоминание/reply, rate-limit, публичная персона
  с раскрытием ИИ), автопостинг по контент-плану, лид-воронка + CRM-выгрузка.
- ВКонтакте (Callback API), WhatsApp (Cloud API) — с той же памятью, склейка персон.

### Мозг
- `stateless-llm` (любой OpenAI-совместимый) и `openclaw` (сессии per-человек, единая
  память в workspace). Несколько инстансов + маршрутизация поверхностей.

### SaaS
- Мультиарендность: `tenant_id` во всех таблицах, изоляция через `AsyncLocalStorage`
  (тесты «A не видит данные B»), миграция старых данных в `default`.
- Тарифы Free/Pro/Enterprise + лимиты, гейт квоты **до** вызова LLM.
- Онбординг: создание арендатора, авто-`setWebhook` по per-tenant секрету, мастер готовности.
- Личный кабинет (self-serve по токену), приём оплаты Robokassa → авто-апгрейд тарифа.
- BYO-LLM для Enterprise (свой endpoint арендатора).

### Безопасность
- Секреты (bot-токены, ключи LLM, токены кабинета) шифруются at-rest (AES-256-GCM),
  поиск по слепому индексу HMAC. Ротация ключа без простоя.
- Timing-safe сравнение секретов; раздельные `API_KEY` / `ADMIN_API_KEY` / токен кабинета.
  В репозитории — только плейсхолдеры.

## Ограничения (честно)

**Блокеры перепродажи как SaaS (для одного владельца — НЕ блокеры):**
1. **Control-plane одноарендный.** `startControlLoop`/автопостинг/дайджест в `server.js`
   запускаются глобально на `ONEINT_BOT_TOKEN`/`CHANNEL_ID`/`OWNER_CHAT_ID` в контексте
   `default`. Входящие маршрутизируются по арендаторам корректно, но ответ владельца на
   кнопку/команду исполнится в контексте `default`, не его арендатора.
2. **Один экземпляр (нет HA).** Long-polling и таймеры очереди — в процессе; две реплики
   конфликтуют по `getUpdates`.

**Осознанные рамки / полировка:**
3. Отправка наружу без ретраев (fallback из персоны, «⚠️ не удалось» владельцу).
4. Кабинет: вход по токену в localStorage (нет полноценного логина/refresh).
5. Robokassa: разовые счета, нет подписок/автосписаний.
6. Ключи шифрования в env (ротация есть), внешнего KMS пока нет.
7. Нет структурного логирования и метрик (`console.*`, нет `/metrics`).
8. Логи содержат тексты переписок (TTL). WhatsApp-диалоги платные.

| Сценарий | Статус |
|---|---|
| Личный / SMB, один владелец | ✅ Production-ready |

| Высокая доступность / несколько реплик | ❌ Не проектировался |

## Чеклист перед запуском (одиночный режим)

- [ ] `API_KEY`, `WEBHOOK_SECRET`, `SECRETS_KEY` заданы; `/api/*` закрыт на nginx по IP; `.env` `chmod 600`.
- [ ] HTTPS-домен; вебхук зарегистрирован (`getWebhookInfo` без ошибок).
- [ ] `/health` → `{"status":"ok","db":true}`; Business-подключение активно.
- [ ] Бэкап `secretary.db` по cron; мониторинг `/health` и рестартов; алерт на `[uncaughtException]`.
- [ ] Прогон в `DRY_RUN=true`, первые дни — `/draft`.

## Дорожная карта мультиарендного SaaS

Полный разбор требований к коммерческому запуску — в эпиках GitHub
(label `saas` + `epic`):

1. **Мультиарендный control-plane** (главный блокер): бот уведомлений/поллинг и
   автопостинг/дайджест на каждого арендатора; исполнение команд в его контексте.
2. **Биллинг: подписки и автопродление** (рекуррентные платежи, грейс-период,
   авто-suspend, фискализация 54-ФЗ).
3. **Масштабирование и надёжность (HA)**: webhook вместо long-polling, очередь в БД с
   воркерами, stateless-реплики, ретраи отправки.
4. **Наблюдаемость и эксплуатация**: структурные логи, метрики `/metrics`, алертинг, аудит-лог.
5. **Безопасность и приватность SaaS**: внешний KMS, полноценный логин в кабинет,
   rate-limiting API, изоляция логов + право на удаление данных (152-ФЗ/GDPR).
6. **Самостоятельный онбординг и продукт**: саморегистрация, тарифная страница, триал,
   email-уведомления, клиентская документация.
